## 🎯 两阶段执行模式

### 第一阶段：内容生成（Phase 1-4）

```
用户请求："帮我写一篇关于 Sentinel 的技术笔记"
  ↓
Phase 1: deep-research 自动调研
  ├─ 调用 deep-research skill
  ├─ 调研技术主题
  └─ 输出结构化素材
  ↓
Phase 2: 内容整理
  ├─ 确定核心场景
  ├─ 组织逻辑结构
  └─ 识别插图位置
  ↓
Phase 3: 场景写作
  ├─ 五步讲解法
  ├─ 代码示例
  └─ 章节总结
  ↓
Phase 4: 智能配图
  ├─ draw.io（复杂架构图）
  ├─ Mermaid（简单流程图）
  └─ AI 图像提示词
      ├─ 生成提示词
      ├─ 写入初稿
      └─ 添加图片描述
  ↓
🔍 Review 功能（新增）
  ├─ 内容逻辑一致性检查
  ├─ 技术准确性检查
  ├─ 场景连贯性检查
  └─ 代码示例完整性检查
  ↓
输出：初稿文档（含图像提示词）
```

### 第二阶段：内容净化（Phase 5-6）

```
用户确认："初稿内容无误，开始清理"
  ↓
Phase 5: 提示词识别和删除
  ├─ 精准识别 [🎨 生图提示词] 区块
  ├─ 仅删除提示词内容
  ├─ 保留图片描述（> 📷 ...）
  └─ 保留最终图像引用
  ↓
Phase 6: 提示词备份沉淀
  ├─ 创建备份文件
  │   路径：resources/images/{文档名}/{文档名}-image-prompts.md
  ├─ 记录提示词原文
  ├─ 记录图像位置
  ├─ 记录图像描述
  └─ 验证备份完整性
  ↓
输出：纯净文档 + 备份文件
```

## 🔄 工作流程详解

### 第一阶段：内容生成

#### Phase 1: deep-research 调研

```markdown
调用 deep-research skill

调研范围：
- 技术定义和原理
- 核心功能和使用场景
- 配置方法和代码示例
- 最佳实践和常见问题
- 与相关技术的对比

输出：结构化素材（约 5000 字）
```

#### Phase 2: 内容整理

```markdown
输入：deep-research 输出素材

处理：
1. 确定核心业务场景（如电商下单）
2. 按"遇到问题 → 引入技术 → 解决问题"组织
3. 识别需要插图的关键位置
4. 准备代码示例和配置片段

输出：场景驱动结构大纲
```

#### Phase 3: 场景写作

```markdown
输入：内容大纲

处理：
按照五步讲解法写作：
① 遇到问题 → 具体困境
② 没有它会怎样 → 朴素方案行不通
③ 一句话定义 + 类比 → 生活比喻
④ 快速上手 → 最小可运行代码
⑤ 深入原理 → 架构图 + 底层流程

输出：完整文档初稿（无插图）
```

#### Phase 4: 智能配图

```markdown
输入：文档初稿

处理：
识别需要插图的位置，按类型生成：

1. 复杂架构图 → draw.io
   - 创建 .drawio XML
   - 导出 PNG
   - 引用到文档

2. 简单流程图 → Mermaid
   - 编写 Mermaid 代码
   - Obsidian 自动渲染

3. 概念插图 → AI 图像提示词
   - 结合上下文生成提示词
   - 写入文档作为初稿标记
   - 添加 20 字图片描述

输出：含插图的完整初稿
```

#### 🔍 Review 功能（新增）

```markdown
输入：完整初稿

检查项：
✅ 内容逻辑一致性
   - 章节间是否有因果递进关系
   - 场景是否贯穿全文
   - 技术引入是否自然

✅ 技术准确性
   - 技术定义是否准确
   - 代码示例是否正确
   - 配置方法是否可行

✅ 场景连贯性
   - 核心场景是否贴切
   - 示例是否符合场景
   - 类比是否恰当

✅ 代码示例完整性
   - 代码是否可运行
   - 是否包含必要注释
   - 是否有完整上下文

输出：Review 报告
- ✅ 通过 → 进入第二阶段
- ❌ 需修改 → 返回 Phase 3 重新写作
```

---

### 第二阶段：内容净化

#### Phase 5: 提示词删除

```markdown
输入：含提示词的初稿

处理：
1. 精准识别所有 [🎨 生图提示词] 区块
   格式：
   [🎨 生图提示词]
   ```
   [提示词内容]
   ```
   > 📷 图片描述

2. 仅删除提示词内容
   - 删除 [🎨 生图提示词] 标记
   - 删除提示词代码块
   - 保留 > 📷 图片描述
   - 保留图像引用

3. 不影响正文内容
   - 章节标题不变
   - 代码示例不变
   - 文字描述不变

输出：纯净文档
```

#### Phase 6: 提示词备份

```markdown
输入：删除的提示词内容

处理：
1. 创建备份文件
   路径：resources/images/{文档名}/{文档名}-image-prompts.md

2. 备份格式：
   # {文档名} - 图片提示词备份
   
   ## 图片 1：[位置描述]
   **位置**：第 X 章第 Y 节（[章节标题]）
   **提示词**：[完整提示词]
   **图片描述**：[20字描述]
   
   ## 图片 2：...

3. 验证备份完整性
   - 检查提示词数量是否匹配
   - 检查格式是否正确
   - 检查文件是否可读写

输出：备份文件（永久保留）
```

## 📊 最佳实践

### 内容生成阶段

1. **明确场景**
   - 指定核心业务场景
   - 场景要贴切、易懂
   - 场景贯穿全文

2. **深度调研**
   - 使用 deep-research 全面调研
   - 覆盖定义、原理、使用、最佳实践
   - 对比相关技术

3. **插图规划**
   - 提前识别需要插图的位置
   - 区分插图类型（架构/流程/概念）
   - 为每种类型选择合适的生成方式

### 内容净化阶段

1. **仔细审阅**
   - 检查内容准确性
   - 检查代码可运行性
   - 检查提示词精确性

2. **备份验证**
   - 确认备份文件创建成功
   - 验证提示词数量匹配
   - 检查备份格式正确

3. **安全删除**
   - 仅删除提示词内容
   - 保留图片描述
   - 保留图像引用
   - 不影响正文
